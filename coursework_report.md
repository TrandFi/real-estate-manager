# 1. Архитектура проекта

В рамках данной курсовой работы спроектировано и разработано веб-приложение **«Real Estate Manager»**, предназначенное для автоматизации процессов публикации, ведения и продажи объектов недвижимости. Разработанная система решает задачи централизованного учета объявлений о продаже недвижимости, отслеживания статусов сделок, управления закрепленными за объектами агентами (риелторами) и ведения истории изменения стадий продажи каждого объекта. 

Информационная система предоставляет пользователям следующие возможности:
*   Регистрация учетных записей и авторизация пользователей в системе;
*   Создание карточек объектов недвижимости с указанием их названия, описания, цены, адреса, площади, ссылки на виртуальный тур и сроков размещения;
*   Интерактивное изменение цены, характеристик объекта и его текущего статуса продажи;
*   Формирование команды агентов для ведения сделки с разграничением внутренних ролей участников («Ведущий агент», «Со-агент», «Покупатель»);
*   Автоматическое журналирование изменения стадий продажи (ведение истории статусов объекта недвижимости);
*   Администрирование прав пользователей и управление активностью учетных записей.

Архитектура разрабатываемого программного комплекса построена по классическому клиент-серверному принципу с жестким разделением зон ответственности (separation of concerns). Приложение состоит из двух функциональных частей:

1.  **Клиентская часть (Frontend)** — разработана на базе стандартов HTML5, CSS3 и современного языка программирования JavaScript (ES6+). Фронтенд функционирует в веб-браузере пользователя по модели Single Page Application (SPA), обеспечивая интерактивное взаимодействие без необходимости перезагрузки страниц.
2.  **Серверная часть (Backend)** — реализована на интерпретируемом языке программирования PHP с использованием объектно-ориентированной библиотеки PDO (PHP Data Objects) для взаимодействия с реляционной системой управления базами данных MySQL.

Взаимодействие между клиентской и серверной частями осуществляется посредством защищенного REST-подобного веб-интерфейса (API). Клиент выполняет асинхронные HTTP-запросы (GET, POST, PUT, DELETE) к серверным скриптам, передавая входные параметры в виде URL-аргументов или структурированного тела запроса. Сервер осуществляет проверку полномочий пользователя, производит операции над данными и возвращает стандартизированные ответы в формате JSON (JavaScript Object Notation), что обеспечивает гибкость интеграции и скорость обмена данными.

*   **`api/properties.php`** — основная точка входа для работы с объектами недвижимости. *   **`api/bootstrap.php`** — выполняет предварительную настройку окружения для каждого API-запроса (инициализация сессии, подключение базы данных, установка заголовков CORS и JSON, глобальные методы разграничения прав доступа).
*   **`config/database.php`** — содержит параметры подключения к СУБД MySQL через драйвер PDO (хост, имя базы данных, учетные данные).


## 1.2. Архитектурный подход

Архитектура веб-приложения «Real Estate Manager» построена на чётком разделении ответственности между клиентской (Frontend) и серверной (Backend) частями. Клиентская часть функционирует полностью в браузере пользователя и не имеет возможности прямого обращения к базе данных или файловой системе. Все действия пользователя (просмотр каталога, отправка поисковых запросов, смена цены, добавление агентов) преобразуются клиентским JavaScript в асинхронные HTTP-запросы к серверному API.

Серверная часть (REST API) принимает поступающие запросы, выполняет аутентификацию пользователя по данным сессии, проверяет права доступа на выполнение запрашиваемой операции в соответствии с ролевой моделью, осуществляет валидацию входных данных, проводит транзакции в базе данных MySQL и возвращает структурированный ответ в формате JSON.

Основные компоненты архитектурного разделения:
*   **Клиентская часть (Frontend)** — отвечает за визуализацию данных и взаимодействие с пользователем. Она динамически формирует и отображает формы авторизации/регистрации, фильтры поиска недвижимости, списки объявлений о продаже объектов, детальную карточку каждого объекта с интерактивным изменением параметров, состав закрепленной команды агентов (риелторов) и специализированную панель администрирования пользователей.
*   **Серверная часть (REST API)** — содержит бизнес-логику системы и обрабатывает входящие запросы в соответствии со стандартами HTTP-методов:
    *   `GET` — безопасные запросы для получения списков недвижимости, детальной информации по объекту, списков пользователей или истории смены статусов;
    *   `POST` — запросы на добавление новых объектов недвижимости, регистрацию или аутентификацию пользователей;
    *   `PUT` — запросы на обновление информации о недвижимости (изменение цены, статуса продажи, редактирование описания) или изменение ролей пользователей;
    *   `DELETE` — запросы на удаление объявлений или отзыв агентов от ведения объекта.
*   **База данных (MySQL)** — является уровнем постоянного хранения данных (Data Storage Layer). Хранит информацию о пользователях, назначенных им ролях, опубликованных объявлениях о продаже недвижимости, статусах объектов, составе закрепленной команды риелторов и детальную историю изменения стадий сделки (историю смены статусов/цен).
*   **Механизм взаимодействия** — обновление графического интерфейса пользователя производится динамически, без перезагрузки всей веб-страницы. Все сетевые транзакции осуществляются асинхронно с помощью встроенного в браузер программного интерфейса `fetch()`, возвращающего Promise-объекты.

Такой архитектурный подход (Decoupled Architecture) обеспечивает ряд существенных преимуществ:
1.  **Масштабируемость и независимая разработка** — визуальное представление (интерфейс) может модифицироваться, перерабатываться и оптимизироваться без изменения серверной бизнес-логики и структуры базы данных.
2.  **Повторное использование API** — разработанная серверная часть может обслуживать не только текущий веб-интерфейс, но и сторонние интеграционные решения, внешние клиенты (например, при тестировании запросов через Postman), а также мобильные или десктопные приложения.
3.  **Повышенная безопасность** — так как база данных скрыта за слоем API, злоумышленник не имеет прямого доступа к СУБД. Все запросы проходят через централизованную точку контроля прав доступа в `bootstrap.php`.

### Особенности разграничения ролевой модели

Ключевой особенностью спроектированной базы данных является строгое разделение **глобальных ролей** пользователей на уровне всей системы (таблицы `roles`, `user_roles`) и их **локальных ролей** в рамках ведения конкретной сделки (таблица `property_agents`).

Такая архитектура дает высокую гибкость:
1. Пользователь с глобальной ролью `agent` (риелтор) в рамках одного объекта недвижимости может выступать в качестве `lead_agent` (ведущего сделку), в другом объекте быть привлеченным в качестве помощника (`co_agent`), а в остальных объектах вообще не иметь прав на редактирование.
2. Глобальная роль `client` (клиент) позволяет пользователю просматривать общую базу объектов, но при закреплении его за конкретным объектом в таблице `property_agents` с ролью `buyer`, он получает авторизованный доступ к просмотру закрытой информации по ходу оформления документов на данный объект.
3. Администраторы (`admin`) имеют глобальный доступ ко всем операциям и могут управлять правами пользователей.

### Реляционные связи и ограничения (Constraints)

Связи между таблицами построены на базе внешних ключей (`FOREIGN KEY`), гарантирующих целостность данных на уровне СУБД:
*   Связь `properties.creator_id` с таблицей `users.id` определяет автора публикации. При удалении пользователя все созданные им карточки автоматически удаляются с помощью каскадного правила `ON DELETE CASCADE`.
*   Связь `properties.status_id` с таблицей `property_statuses.id` обеспечивает жесткое ограничение на использование только предопределенных стадий продажи.
*   Таблицы связей `property_agents` и `user_roles` используют составные первичные ключи (`PRIMARY KEY`), исключающие дублирование записей, а также каскадное удаление связей при удалении родительских сущностей.
*   История изменений в таблице `property_status_history` сохраняется с фиксацией изменившего статус пользователя (`changed_by`), что позволяет проводить полный аудит действий в системе.

### SQL-скрипт инициализации базы данных

Для развертывания структуры базы данных на сервере MySQL используется следующий SQL-скрипт (DDL):

```sql
-- 1. Таблица глобальных ролей
CREATE TABLE roles (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 2. Таблица пользователей
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    fullname VARCHAR(255) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    meta_info VARCHAR(255) DEFAULT NULL, -- Хранит контактный телефон или название агентства
    is_active TINYINT(1) DEFAULT 1
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 3. Таблица связи пользователей с глобальными ролями (многие-ко-многим)
CREATE TABLE user_roles (
    user_id INT NOT NULL,
    role_id INT NOT NULL,
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 4. Таблица стадий продажи недвижимости
CREATE TABLE property_statuses (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL UNIQUE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 5. Таблица объявлений объектов недвижимости
CREATE TABLE properties (
    id INT AUTO_INCREMENT PRIMARY KEY,
    title VARCHAR(255) NOT NULL,
    description TEXT,
    creator_id INT NOT NULL,
    status_id INT NOT NULL,
    progress_percent INT DEFAULT 0, -- Прогресс подготовки документов к сделке (0-100%)
    start_date DATE NOT NULL,
    planned_end_date DATE,
    repository_link VARCHAR(255) DEFAULT NULL, -- Ссылка на виртуальный тур/фотогалерею
    FOREIGN KEY (creator_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (status_id) REFERENCES property_statuses(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 6. Таблица связи агентов и участников сделки с объектами
CREATE TABLE property_agents (
    property_id INT NOT NULL,
    user_id INT NOT NULL,
    role VARCHAR(50) NOT NULL, -- lead_agent, co_agent, buyer
    PRIMARY KEY (property_id, user_id),
    FOREIGN KEY (property_id) REFERENCES properties(id) ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- 7. Таблица истории изменения статусов недвижимости
CREATE TABLE property_status_history (
    id INT AUTO_INCREMENT PRIMARY KEY,
    property_id INT NOT NULL,
    status_id INT NOT NULL,
    changed_by INT NOT NULL,
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (property_id) REFERENCES properties(id) ON DELETE CASCADE,
    FOREIGN KEY (status_id) REFERENCES property_statuses(id),
    FOREIGN KEY (changed_by) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```



# 2. Документация к API

В рамках курсовой работы был спроектирован и реализован защищенный веб-интерфейс программирования приложений (API) для предметной области «управление сделками с недвижимостью». Веб-API реализует полный набор CRUD-операций (создание, чтение, обновление, удаление) для объектов недвижимости, управления составом риелторов и покупателей, а также дополнительные операции для авторизации, сессионного контроля и администрирования.

Все серверные скрипты возвращают данные в едином формате JSON. В случае успешного завершения запроса возвращается параметр `"success": true` и запрошенные данные; при возникновении ошибки возвращается `"success": false` и описание ошибки в строковом свойстве `"message"`.

Серверная часть функционирует на базе прозрачного адаптационного слоя: она принимает и отдает параметры, ожидаемые исходным JavaScript-кодом клиентской части (проекты, группы, роли «student»/«teacher»), но производит операции над таблицами новой базы данных недвижимости (объявления `properties`, контактные данные `meta_info`, роли `client` (покупатель) / `agent` (риелтор)).

---

## 2.1. Базовый модуль инициализации и безопасности (api/bootstrap.php)

Модуль `bootstrap.php` служит единой точкой предварительной конфигурации серверной среды (Middleware). Он автоматически подключается во все остальные скрипты API в первой строке выполнения.

### Назначение модуля:
1. **Инициализация сессий**: Вызывает стандартный механизм `session_start()` для управления куки-сессиями пользователей.
2. **Установка системных заголовков**: Задает тип контента как UTF-8 JSON (`Content-Type: application/json`) и прописывает заголовки CORS (`Access-Control-Allow-Origin: *` и др.) для отладки.
3. **Глобальное подключение к БД**: Подгружает файл конфигурации PDO `config/database.php` и создает экземпляр `$db`.
4. **Вспомогательные методы**:
   * `sendResponse($success, $message, $extra, $code)` — общая функция завершения запроса и формирования JSON-ответа с соответствующим HTTP-кодом.
   * `getCurrentUser($db)` — извлекает данные и глобальные роли авторизованного пользователя.
   * `requireAuth($db)` — блокирует выполнение скрипта с отправкой 401 HTTP-кода, если сессия отсутствует.
   * `isAdmin($user)` / `isAgent($user)` — хелперы ролевой модели.
   * `getJsonInput()` — считывает входящий поток данных JSON из тела запроса (`php://input`).

### Ключевые фрагменты кода:
```php
function sendResponse($success, $message = '', $data = [], $statusCode = 200) {
    http_response_code($statusCode);
    $response = array_merge([
        'success' => $success
    ], $data);
    if (!empty($message)) {
        $response['message'] = $message;
    }
    echo json_encode($response, JSON_UNESCAPED_UNICODE);
    exit;
}

function getCurrentUser($db) {
    if (!isset($_SESSION['user_id'])) {
        return null;
    }
    try {
        $stmt = $db->prepare("
            SELECT id, fullname, email, meta_info, is_active 
            FROM users 
            WHERE id = ? AND is_active = 1
        ");
        $stmt->execute([$_SESSION['user_id']]);
        $user = $stmt->fetch();
        if (!$user) return null;
        
        $stmtRoles = $db->prepare("
            SELECT r.name FROM roles r
            JOIN user_roles ur ON ur.role_id = r.id
            WHERE ur.user_id = ?
        ");
        $stmtRoles->execute([$user['id']]);
        $user['roles'] = $stmtRoles->fetchAll(PDO::FETCH_COLUMN);
        return $user;
    } catch (PDOException $e) {
        return null;
    }
}
```

## 2.4. Модуль администрирования пользователей (api/users.php)

Модуль предназначен для администрирования глобальных профилей пользователей. Доступ к методам изменения ролей и деактивации аккаунтов разрешен только администраторам.

### Описание конечных точек (Эндпоинтов):

| Метод | URL | Назначение |
| :--- | :--- | :--- |
| **GET** | `api/users.php` | Запрос списка всех зарегистрированных пользователей системы и их глобальных ролей. |
| **GET** | `api/users.php?id={id}` | Получение системной информации об одном конкретном пользователе. |
| **PUT** | `api/users.php?id={id}` | Обновление глобальных ролей пользователя и его флага активности `is_active` (доступно только Администраторам). |

### Подробности реализации:
1. **Авторизационный барьер**: При вызове метода `PUT` скрипт проверяет глобальную роль авторизованного пользователя с помощью метода `isAdmin($user)`. Обычные клиенты и риелторы получают ответ с кодом 403 Forbidden.
2. **Защита от самоблокировки**: Скрипт осуществляет сравнение ID изменяемого профиля с ID текущего администратора из сессии. Запрещается деактивировать самого себя или снимать с себя роль `admin` во избежание потери доступа ко всей панели администрирования.
3. **Обновление ролей**: Поскольку у пользователя может быть несколько глобальных ролей (связь «многие-ко-многим»), при изменении ролей скрипт сначала очищает старые записи в таблице `user_roles`, а затем вносит актуальную связь в рамках транзакции.

### Ключевые фрагменты кода:
```php
// Безопасность в api/users.php (PUT)
if (!isAdmin($user)) {
    sendResponse(false, 'У вас нет прав администратора для совершения этой операции.', [], 403);
}
if ($id === $user['id']) {
    sendResponse(false, 'Вы не можете изменить статус или права собственной учетной записи.');
}

try {
    $db->beginTransaction();
    // Обновляем активность
    $stmtUpdateUser = $db->prepare("UPDATE users SET is_active = ? WHERE id = ?");
    $stmtUpdateUser->execute([$isActive, $id]);
    
    // Обновляем роли
    if (!empty($targetRoles)) {
        $stmtDelRoles = $db->prepare("DELETE FROM user_roles WHERE user_id = ?");
        $stmtDelRoles->execute([$id]);
        
        foreach ($targetRoles as $feRole) {
            $dbRoleName = mapFrontendRoleToDb($feRole);
            $stmtGetRole = $db->prepare("SELECT id FROM roles WHERE name = ?");
            $stmtGetRole->execute([$dbRoleName]);
            $roleId = $stmtGetRole->fetchColumn();
            
            if ($roleId) {
                $stmtInsRole = $db->prepare("INSERT INTO user_roles (user_id, role_id) VALUES (?, ?)");
                $stmtInsRole->execute([$id, $roleId]);
            }
        }
    }
    $db->commit();
} catch (PDOException $e) {
    if ($db->inTransaction()) $db->rollBack();
    sendResponse(false, 'Ошибка БД: ' . $e->getMessage());
}
```
